# Dual-Runtime Claude Code Plugin — Design Specification

## Summary

Transform `maestro-gemini` into `maestro-orchestrate` — a dual-runtime (Gemini CLI + Claude Code) multi-agent orchestration platform with its own MCP server source, no dependency on overture. Three phases, each independently shippable with a clean rollback path.

## Context

**Maestro** is a multi-agent orchestration platform currently shipping as a Gemini CLI extension. It coordinates 22 specialized subagents through two workflows: Express (simple tasks) and Standard 4-phase (medium/complex tasks). The platform was originally generated by **overture**, a multi-runtime generator framework that transforms canonical agent definitions into runtime-specific packages for Gemini CLI, Claude Code, and Codex.

Overture already generates a complete Claude Code plugin (`dist/claude-plugin/`) from canonical content, but the generator has proven to be an unnecessary layer of indirection. This spec defines the path to:

1. Bring the Claude Code plugin into this repo as a first-class runtime
2. Migrate the MCP server source from overture
3. Replace overture's 688-line generator with a lightweight in-repo script
4. Rebrand from `maestro-gemini` to `maestro-orchestrate`
5. Decommission overture

**Overture reference commit**: `b314d9d` (the version that generated the current maestro-gemini content).

## Constraints

- All work happens within this repository on the `main` branch
- No symlinks — must work across all OS and policy configurations
- Each phase is independently shippable and rollback-safe
- Existing Gemini CLI functionality must not regress at any point
- The user-facing extension/plugin name stays `maestro` on both platforms

## Cross-Cutting Concerns

### Claude Code Hook I/O Contract

Claude Code hooks use a different I/O contract than Gemini hooks. The `hook-adapter.js` in `claude/scripts/` must normalize between these:

| Aspect | Gemini CLI | Claude Code |
|--------|-----------|-------------|
| Hook events | `SessionStart`, `BeforeAgent`, `AfterAgent`, `SessionEnd` | `SessionStart`, `SessionEnd`, `PreToolUse`, `PostToolUse` |
| PreToolUse input | N/A | `{ tool_name, tool_input }` via stdin JSON |
| Denial signaling | `{ continue: false, systemMessage: "..." }` | `process.exit(2)` + `permissionDecision: "deny"` in stdout JSON |
| Path variable | `${extensionPath}` | `${CLAUDE_PLUGIN_ROOT}` |
| Agent detection | `BeforeAgent` event with `prompt` field | `PreToolUse` event with matcher `Agent`, `tool_input` contains agent prompt |

Overture's generated `hook-adapter.js` and `policy-enforcer.js` (commit `b314d9d`) implement the Claude-specific contract: `policy-enforcer.js` uses `process.exit(2)` for deny decisions and reads `tool_input.command` for Bash tool calls. These files are copied as-is in Phase A. Functional validation against a live Claude Code session is a Phase A acceptance criterion.

### Environment Variable Resolution

Gemini CLI sets `MAESTRO_WORKSPACE_PATH` (via `gemini-extension.json` mcpServers env) and makes `${extensionPath}` available. Claude Code provides `${CLAUDE_PLUGIN_ROOT}` and `${CLAUDE_PROJECT_DIR}` instead. The `lib/config/setting-resolver.js` and `lib/core/project-root-resolver.js` read Gemini-specific env vars.

The `claude/lib/` copy must be patched to also check `CLAUDE_PLUGIN_ROOT` (as fallback for `MAESTRO_EXTENSION_PATH`) and `CLAUDE_PROJECT_DIR` (as fallback for `MAESTRO_WORKSPACE_PATH`). This is a targeted 2-line addition per file — not a rewrite. Phase A must include these patches; otherwise, settings resolution and project root detection will silently fall back to less reliable heuristics under Claude Code.

## Phase A: Claude Code Static Integration

### Objective

Add Claude Code plugin artifacts to this repo as static files, colocated with existing Gemini extension files. Zero modifications to existing files. Purely additive.

### Core Problem

Both Gemini CLI and Claude Code discover agents from `agents/*.md` and skills from `skills/*/SKILL.md` relative to their extension/plugin root. The two runtimes require incompatible content:

| Difference | Gemini CLI | Claude Code |
|------------|-----------|-------------|
| Agent frontmatter | `kind: local`, `temperature`, `max_turns`, `timeout_mins` | `model: inherit`, `color`, example triggers in description |
| Tool names | `read_file`, `write_file`, `run_shell_command`, `ask_user` | `Read`, `Write`, `Bash`, `AskUserQuestion` |
| Agent filenames | `snake_case` (`code_reviewer.md`) | `kebab-case` (`code-reviewer.md`) |
| Command format | TOML (`commands/maestro/*.toml`) | Markdown (`commands/*.md` with `allowed-tools` frontmatter) |
| Hook schema | Gemini hook events (`SessionStart`, `BeforeAgent`, `AfterAgent`) | Claude hook events (`SessionStart`, `SessionEnd`, `PreToolUse`) |
| Orchestrator | `GEMINI.md` context file | Embedded in command Markdown + command-backed skill |
| Policies | `policies/maestro.toml` (Gemini policy engine) | `scripts/policy-enforcer.js` (PreToolUse hook) |

If both runtimes point to the same root, they discover each other's incompatible content.

### Solution: Nested Plugin Root

Repo root serves as the Gemini extension root (unchanged). A `claude/` subdirectory serves as the Claude Code plugin root. Each runtime discovers only its own content.

Shared resources (`lib/`, `mcp/`, `templates/`, `references/`) are **fully copied** into `claude/` — no symlinks. This duplication is intentional and temporary; Phase C eliminates it.

All files under `claude/` are copied from overture's `dist/claude-plugin/` output at commit `b314d9d`. The generated output works as-is because `claude/scripts/*.js` require from `../lib/` which resolves to `claude/lib/` (the full copy). Two files in `claude/lib/` receive targeted patches for Claude Code env var fallbacks (see Modifications to Existing Files).

| Directory | Files | Change Frequency | Risk | Mitigation |
|-----------|-------|-----------------|------|------------|

```bash

```

### Modifications to Existing Files

Phase A is predominantly additive. The only modifications:

| File | Change | Reason |
|------|--------|--------|
| `.geminiignore` | Add `claude/` | Prevent Gemini CLI from indexing Claude-specific content into its context window |
| `claude/lib/config/setting-resolver.js` | Add `CLAUDE_PLUGIN_ROOT` fallback for `MAESTRO_EXTENSION_PATH` | Settings resolution under Claude Code (see Cross-Cutting Concerns) |
| `claude/lib/core/project-root-resolver.js` | Add `CLAUDE_PROJECT_DIR` fallback for `MAESTRO_WORKSPACE_PATH` | Project root detection under Claude Code |

No existing Gemini-runtime files are modified. `package.json` `files` array is intentionally NOT updated — the npm package is for Gemini extension distribution only; Claude plugin distribution uses a separate mechanism.

### Installation

**Gemini CLI** (unchanged):
```bash
gemini extensions install https://github.com/josstei/maestro-gemini
```

**Claude Code** (new):
Register the `claude/` subdirectory as a Claude Code plugin. The exact installation mechanism depends on Claude Code's plugin registration (manual `installed_plugins.json` entry or future `claude plugins install` command). The plugin root path must resolve to `<repo>/claude/`. MCP server is configured separately — copy `claude/mcp-config.example.json` into the user's Claude Code MCP settings, adjusting the `cwd` to the repo root.

### Validation Criteria

1. `gemini extensions list` still shows maestro (no Gemini regression)
2. Claude Code discovers 22 agents from `claude/agents/`
3. Claude Code discovers 18 skills from `claude/skills/`
4. Hook scripts execute without `MODULE_NOT_FOUND` errors
5. `/maestro-orchestrate` command activates in Claude Code
6. MCP server starts and responds to `get_session_status`
7. `policy-enforcer.js` blocks `rm -rf` via PreToolUse hook (exits with code 2, returns `permissionDecision: "deny"`)
8. Claude lib env var fallbacks resolve correctly (`CLAUDE_PLUGIN_ROOT` for extension path, `CLAUDE_PROJECT_DIR` for workspace path)

### Rollback

```bash
rm -rf claude/
```

Zero impact on Gemini functionality.

## Phase B: MCP Server Source Migration

### Objective

Move the unbundled MCP server source from overture into this repo. Add a build step that produces the bundled output. Both runtimes reference the same built artifact — eliminating the `claude/mcp/` duplication from Phase A.

### MCP Source (from overture `plugins/maestro/src/mcp/`)

```
src/mcp/
├── maestro-server.js              # Server bootstrap, tool registration, dispatch
├── handlers/
│   ├── initialize-workspace.js    # Workspace directory creation
│   ├── assess-task-complexity.js  # Repo signal analysis for complexity classification
│   ├── validate-plan.js           # Acyclic dep validation, file ownership, agent validity
│   └── session-state-tools.js     # create_session, update_session, transition_phase,
│                                  # get_session_status, archive_session, resolve_settings
└── roots-discovery.js             # Workspace root resolution (MCP roots/list → env → git → cwd)
```

Dependencies bundled by esbuild: `@modelcontextprotocol/sdk`, `zod`, and imports from `lib/state/` and `lib/core/`.

### Changes

**New files:**

| File | Source |
|------|--------|
| `src/mcp/maestro-server.js` | overture `plugins/maestro/src/mcp/maestro-server.js` |
| `src/mcp/handlers/*.js` (4 files) | overture `plugins/maestro/src/mcp/handlers/` |
| `src/mcp/roots-discovery.js` | overture `plugins/maestro/src/mcp/roots-discovery.js` |

**Modified files:**

| File | Change |
|------|--------|
| `package.json` | Add `devDependencies.esbuild`, add `scripts.build:mcp` |
| `claude/mcp-config.example.json` | Update path to `../mcp/maestro-server.js` (points to repo root) |

**Deleted files:**

| File | Reason |
|------|--------|
| `claude/mcp/maestro-server.js` | Both runtimes now share `mcp/maestro-server.js` at repo root |

**Build command:**
```json
{
  "scripts": {
    "build:mcp": "esbuild src/mcp/maestro-server.js --bundle --platform=node --target=node18 --format=cjs --outfile=mcp/maestro-server.js"
  },
  "devDependencies": {
    "esbuild": "^0.25.0"
  }
}
```

### Import Path Adjustments

MCP handler source in overture imports from `../../lib/` relative to `plugins/maestro/src/mcp/handlers/`. In the new layout at `src/mcp/handlers/`, imports reach `lib/` at repo root:

| Import in overture | Import in this repo |
|-------------------|---------------------|
| `../../lib/state/session-state` | `../../../lib/state/session-state` |
| `../../lib/core/logger` | `../../../lib/core/logger` |
| `../../lib/core/atomic-write` | `../../../lib/core/atomic-write` |

Mechanical find-replace. These paths are resolved at esbuild bundle time only — not present in the output.

### MCP Registration (both runtimes, single artifact)

**Gemini CLI** (`gemini-extension.json` — unchanged):
```json
{ "mcpServers": { "maestro": { "command": "node", "args": ["${extensionPath}/mcp/maestro-server.js"] } } }
```

**Claude Code** (`claude/mcp-config.example.json` — updated):
```json
{ "maestro": { "command": "node", "args": ["mcp/maestro-server.js"], "cwd": "<path-to-repo-root>" } }
```

### Bundled Output: Committed

Both runtimes install without a build step. Users never run `npm run build:mcp`. The build command is for developers modifying MCP source.

### Validation Criteria

1. `npm run build:mcp` produces `mcp/maestro-server.js` with identical tool surface
2. Gemini MCP tools (`resolve_settings`, `get_session_status`, etc.) respond correctly
3. Claude MCP connection works with updated config path
4. No runtime `require` of unbundled dependencies (all inlined by esbuild)
5. `claude/mcp/` directory no longer exists

### Rollback

Revert `package.json`, delete `src/mcp/`, restore `claude/mcp/` from overture dist. The committed `mcp/maestro-server.js` was never overwritten — same file.

## Phase C: Lightweight Generator + Rebrand

### Objective

Eliminate all content duplication between runtimes by extracting canonical content and building a lightweight in-repo generator. Rebrand from `maestro-gemini` to `maestro-orchestrate`.

### Problem

After Phase A+B, duplicated content:

| Content | Gemini copy | Claude copy | Files |
|---------|------------|-------------|-------|
| Agents (persona/methodology) | `agents/` (22) | `claude/agents/` (22) | 22 pairs |
| Skills (core) | `skills/` (7) | `claude/skills/` (7 shared + 11 Claude-only) | 7 pairs |
| Lib | `lib/` (14) | `claude/lib/` (14, identical except 2 patched) | 14 files |
| Templates | `templates/` (3) | `claude/templates/` (3, identical) | 3 identical |
| References | `references/` (1) | `claude/references/` (1, identical) | 1 identical |

Agent bodies are ~95% identical between runtimes. Differences are mechanical: tool names, path variables, frontmatter schema, filename casing.

### Generator Scope

**Does:**
- Token replacement: `{{tool:file_read}}` → `read_file` (Gemini) / `Read` (Claude)
- Path replacement: `{{path:plugin_root}}` → `${extensionPath}` / `${CLAUDE_PLUGIN_ROOT}`
- Frontmatter injection: canonical base merged with runtime-specific overrides
- Filename transformation: snake_case (Gemini) / kebab-case (Claude)
- Copy shared files (`lib/`, `templates/`, `references/`, `mcp/`, `scripts/`) into each output

**Does not (scope guard — if the script exceeds 500 lines including `js-yaml` usage, stop and reassess):**
- Body assembly with injection ordering
- Template inclusion (`{{include: ...}}`)
- Cross-plugin support
- Emitter abstraction layer
- Vocabulary validation

**Dependencies:** `js-yaml` for frontmatter parsing/serialization. Frontmatter contains multi-line strings, arrays, and nested objects that cannot be reliably parsed with regex. The line count cap applies to `generate.js` only, not the library.

### Content That Is NOT Canonical

The following content is fundamentally different between runtimes and lives exclusively in `runtimes/<runtime>/entry-points/` — the generator copies them as-is with no token replacement:

- **Orchestrator**: `GEMINI.md` (409 lines, Gemini-specific tool interactions, path variables, constraints, delegation via native subagent calls) vs `maestro-orchestrate.md` + `maestro-orchestrate-skill.md` (Claude-specific tool names, `Agent` tool delegation, `AskUserQuestion` interactions). These are structurally incompatible documents — not candidates for token replacement.
- **11 command-backed skills**: `maestro-review/SKILL.md`, `maestro-debug/SKILL.md`, etc. These exist in Claude only (Gemini uses TOML commands + skill activation instead). They live in `runtimes/claude/entry-points/` and are copied directly to `dist/claude/skills/`.
- **Gemini TOML commands**: 12 TOML files in `runtimes/gemini/entry-points/` — no Claude equivalent (Claude uses Markdown commands).
- **Hook scripts**: `claude/scripts/*.js` have Claude-specific I/O adapters; `hooks/*.js` (Gemini) have Gemini-specific adapters. Both live in their respective `runtimes/<runtime>/` directories.

### Canonical Content Layout

```
│   ├── agents/                          # 22 agent files with {{tool:*}} and {{path:*}} tokens
│   ├── skills/                          # 7 core skills with tokens
│   └── orchestrator/                    # Shared orchestrator workflow logic
├── runtimes/
│   ├── gemini/
│   │   ├── name-map.json               # Token → Gemini names
│   │   ├── agents/*.json               # Per-agent frontmatter overrides
│   │   ├── entry-points/               # TOML commands, GEMINI.md
│   │   └── hooks.json
│   └── claude/
│       ├── name-map.json               # Token → Claude names
│       ├── agents/*.json               # Per-agent frontmatter overrides
│       ├── entry-points/               # Markdown commands, command-backed skills
│       └── hooks.json
├── generator/
│   └── generate.js                     # ~200-300 lines
├── lib/                                # Shared library

├── mcp/                                # Built MCP output
├── scripts/                            # Shared utility scripts
├── templates/                          # Shared document templates
└── references/                         # Shared architecture docs
```

### Name Map Format

Each runtime has a `name-map.json` with three sections:

**`tools`**: Maps abstract tool tokens to runtime-specific names.
```json
{
  "file_read": "Read",
  "file_write": "Write",
  "shell_exec": "Bash",
  "user_prompt": "AskUserQuestion"
}
```

**`paths`**: Maps abstract path tokens to runtime-specific variables.
```json
{
  "plugin_root": "${CLAUDE_PLUGIN_ROOT}",
  "state_dir": "docs/maestro",
  "config_home": "~/.claude"
}
```

**`naming`**: Controls filename and format conventions.
```json
{
  "agent_filename": "kebab-case",
  "command_format": "md"
}
```

### Canonical Agent Example

```markdown
---
name: coder
description: |
  Implementation specialist for writing clean, well-structured code...
---

You are a **Senior Software Engineer**...

**Methodology:**
- Use {{tool:file_read}} to read existing code before writing
- Use {{tool:file_write}} and {{tool:file_edit}} for modifications
- Run validation via {{tool:shell_exec}}
```

The generator reads this, loads the runtime's `agents/coder.json` override (tools array, model, temperature, etc.), merges frontmatter, replaces tokens, transforms filename, and writes the output.

### Generator Algorithm

```
1. Parse --runtime argument (gemini | claude | both)
2. Load name-map.json for target runtime(s)
3. For each agent in canonical/agents/:
   a. Parse frontmatter + body
   b. Load runtime override from runtimes/<runtime>/agents/<name>.json
   c. Merge: canonical frontmatter ← runtime override (override wins)
   d. Replace all {{tool:*}} and {{path:*}} tokens in body
   e. Transform filename per naming.agent_filename
   f. Write to dist/<runtime>/agents/
4. For each skill in canonical/skills/:
   a. Read all .md files in skill directory
   b. Replace tokens in each .md
   c. Write to dist/<runtime>/skills/
5. Copy entry-points from runtimes/<runtime>/entry-points/
6. Copy hooks config from runtimes/<runtime>/hooks.json
7. Copy shared directories: lib/, scripts/, templates/, references/, mcp/
8. Write runtime manifest
9. Validate: scan dist/ for any remaining {{tool:*}} or {{path:*}} tokens — fail if found
```

### Generated Output

`npm run generate` produces `dist/gemini/` and `dist/claude/`, each a complete self-contained runtime package.

### `dist/` Commit Strategy

`dist/` is **committed to git** (remove `dist/` from `.gitignore`). Rationale:
- Users install without a build step (`gemini extensions install` or Claude plugin registration)
- GitHub releases reference repo content directly
- CI regenerates and verifies `dist/` matches — fails if stale

To prevent direct edits to generated output, CI validates that `npm run generate` produces no diff against committed `dist/`.

### Installation Migration (Breaking Change)

Phase C changes the Gemini installation path from `gemini extensions link .` (repo root) to `gemini extensions link dist/gemini/` (generated output). This is a **breaking change** for existing users.

Migration strategy:
1. The GitHub repo rename (`maestro-gemini` → `maestro-orchestrate`) already forces users to update their install command (GitHub auto-redirects the old URL, but `gemini extensions install` may not follow redirects)
2. `README.md` documents the new installation command prominently
3. `CHANGELOG.md` includes a "Breaking Changes" section with migration instructions
4. The previous release tag (`v1.4.0`) remains installable from the old URL for users who pin versions
5. Consider adding a `gemini-extension.json` stub at repo root that errors with a message directing users to `dist/gemini/`

### Rebrand

| File | Change |
|------|--------|
| `package.json` | `name` → `@maestro-orchestrator/maestro-orchestrate` |
| `gemini-extension.json` | `name` stays `maestro` (user-facing name) |
| `.claude-plugin/plugin.json` | `name` stays `maestro` (user-facing name) |

|------|--------|
